The VRChat Role Icons system uses object pooling to efficiently manage icon GameObjects, eliminating the performance cost of constantly creating and destroying instances.

## Why Object Pooling?

Without pooling, creating and destroying GameObjects for 80 players would cause:
- Expensive `Instantiate()` and `Destroy()` calls
- Frequent garbage collection spikes
- Unstable memory usage
- Performance degradation over time

**With pooling**, GameObjects are reused, providing:
- No runtime `Instantiate()`/`Destroy()` operations
- Reduced garbage collection by ~66%
- Stable memory footprint (~5 MB vs ~15 MB)
- Consistent performance

## Pool Architecture

The pooling system in `UserIconManager.cs` uses two parallel arrays:

```csharp UserIconManager.cs:32-36
private GameObject[] iconPool;    // Reusable icon pool
private int[] poolInUse;          // Pool state (0=free, 1=in use)
private int poolSize = 0;         // Current pool size

[SerializeField] private int maxIconPool = 20; // Maximum active icons
```

### Pool State Tracking

- `iconPool[]` - Stores references to pooled GameObjects
- `poolInUse[]` - Tracks availability (0 = free, 1 = in use)
- `poolSize` - Current number of icons in the pool
- `maxIconPool` - Maximum pool capacity (configurable)

## Pool Lifecycle

### Initialization

The pool is initialized at startup with empty arrays:

```csharp UserIconManager.cs:60-63
iconPool = new GameObject[maxIconPool];
poolInUse = new int[maxIconPool];
```

Icons are created on-demand rather than pre-instantiated, saving initial load time.

### Getting an Icon from Pool

When a player needs an icon, the system first attempts to reuse an existing one:

```csharp UserIconManager.cs:352-363
private GameObject GetIconFromPool()
{
    for (int i = 0; i < poolSize; i++)
    {
        if (poolInUse[i] == 0 && iconPool[i] != null)
        {
            poolInUse[i] = 1;  // Mark as in use
            return iconPool[i];
        }
    }
    return null;  // No free icons available
}
```

**Algorithm**:
1. Iterate through existing pool entries
2. Find first icon marked as free (`poolInUse[i] == 0`)
3. Mark it as in use (`poolInUse[i] = 1`)
4. Return the GameObject reference
5. Return `null` if no free icons exist

### Creating New Icons

If no free icons exist and the pool isn't full, create a new one:

```csharp UserIconManager.cs:365-380
private GameObject CreateNewIcon()
{
    if (iconPrefab == null || poolSize >= maxIconPool) return null;
    
    GameObject newIcon = Instantiate(iconPrefab);
    if (newIcon != null)
    {
        iconPool[poolSize] = newIcon;   // Add to pool
        poolInUse[poolSize] = 1;        // Mark as in use
        poolSize++;                      // Increment pool size
        newIcon.SetActive(false);        // Start deactivated
        return newIcon;
    }
    
    return null;
}
```

**Process**:
1. Check if pool has room (`poolSize < maxIconPool`)
2. Instantiate new GameObject from prefab
3. Add to pool at current `poolSize` index
4. Mark as in use
5. Increment `poolSize` counter
6. Return deactivated GameObject

### Returning Icons to Pool

When a player disconnects or loses their role, their icon is returned to the pool:

```csharp UserIconManager.cs:382-403
private void ReturnIconToPool(GameObject icon)
{
    if (icon == null) return;
    
    // Clear the icon
    IconFollower follower = icon.GetComponent<IconFollower>();
    if (follower != null)
    {
        follower.CleanupIcon();  // Reset internal state
    }
    
    icon.SetActive(false);  // Deactivate GameObject
    
    // Mark as free in pool
    for (int i = 0; i < poolSize; i++)
    {
        if (iconPool[i] == icon)
        {
            poolInUse[i] = 0;  // Mark as free
            break;
        }
    }
}
```

**Cleanup steps**:
1. Call `CleanupIcon()` to reset IconFollower state
2. Deactivate the GameObject
3. Find the icon's index in the pool
4. Mark as free (`poolInUse[i] = 0`)

## Pool Usage in Practice

When assigning an icon to a player, the system tries pool first:

```csharp UserIconManager.cs:316-330
GameObject icon = GetIconFromPool();

if (icon == null)
{
    // If there are no icons available in the pool and we haven't reached the maximum
    if (poolSize < maxIconPool)
    {
        icon = CreateNewIcon();
    }
    else
    {
        DebugLog("Icon pool is full, no more can be created");
        return;
    }
}
```

**Decision tree**:
1. Try to get free icon from pool
2. If none available and pool not full → create new icon
3. If pool is full → skip this player (they won't get an icon)

## Pool Limiting Strategy

The `maxIconPool` parameter limits simultaneous active icons:

```csharp
[SerializeField] private int maxIconPool = 20;
```

### Recommended Settings

| Player Count | Max Icon Pool | Why |
|--------------|---------------|-----|
| 20-40 players | 15 | Casual worlds, moderate density |
| 40-60 players | 20 | Crowded worlds, balanced performance |
| 60-80 players | 20 | Massive worlds, strict limit needed |

<Note>
**Pool Full Behavior**: When the pool is full, players beyond the limit won't receive icons until others disconnect or move out of range. This is an intentional trade-off to maintain performance.
</Note>

## Performance Benefits

### Memory Stability

| Metric | Without Pooling | With Pooling (20 max) | Improvement |
|--------|----------------|----------------------|-------------|
| GameObjects Active | 80 | 20 | -75% |
| Memory Usage | ~15 MB | ~5 MB | -66% |
| GC Allocations | High variance | Stable | Consistent |

### CPU Efficiency

**Without pooling** (per player join/leave):
- `Instantiate()`: ~2-5ms
- `Destroy()`: ~1-3ms
- Garbage collection spikes: ~10-20ms

**With pooling** (per player join/leave):
- `GetIconFromPool()`: ~0.01ms
- `ReturnIconToPool()`: ~0.01ms
- No GC spikes

## Monitoring Pool Usage

Enable debug logging to monitor pool behavior:

```csharp
[SerializeField] private bool enableDebugLogs = true;
```

Add custom monitoring in `UserIconManager`:

```csharp
Debug.Log($"Pool Status: {poolSize}/{maxIconPool} icons created, {activeIcons.Count} active");
```

**Healthy pool metrics**:
- `poolSize` should stabilize after initial players join
- `activeIcons.Count` should match visible players with roles
- Gap between `poolSize` and `maxIconPool` indicates room to grow

<Warning>
If you see "Icon pool is full" messages frequently, consider:
- Increasing `maxIconPool`
- Reducing `maxDistance` to hide distant icons
- Increasing `distanceCheckInterval` for faster icon recycling
</Warning>

## Best Practices

<AccordionGroup>
  <Accordion title="Set pool size based on world capacity">
    Don't set `maxIconPool` to match max players. Use 25-30% of world capacity:
    - 80 player world → 20 icon pool
    - 40 player world → 15 icon pool
    - 20 player world → 10 icon pool
  </Accordion>

  <Accordion title="Monitor pool efficiency during testing">
    Test with varying player counts and monitor:
    - How quickly the pool fills
    - Whether players are getting skipped
    - Memory usage over time
  </Accordion>

  <Accordion title="Balance pool size with distance culling">
    Smaller pool sizes work well with aggressive distance culling:
    - Small pool (10) + short distance (20m) = Very efficient
    - Large pool (30) + long distance (50m) = More players see icons but higher cost
  </Accordion>
</AccordionGroup>
